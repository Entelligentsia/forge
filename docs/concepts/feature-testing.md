# Feature Testing

Forge testing is heavily structured to align with the framework's entity model. Instead of sprawling test suites, testing is categorized into a three-layer model, ensuring deterministic validation of capabilities.

## Three-Layer Test Model

1. **Task Acceptance:** Fast, localized assertions proving the atomic change generated by a task fulfills its objective (often manual or simple assertions like `node --check`).
2. **Sprint Smoke:** Integration tests run at the conclusion of a sprint to ensure the sprint's batch of tasks did not break adjacent boundaries.
3. **Feature Regression:** Expensive, comprehensive tests mapping to specific [Features](feature.md), asserting that the core business domain described by the feature holds true over time.

## Identifier Linking (FEAT-NNN)

To enforce traceability back to the architecture, all files or named suites in the feature regression layer must carry the **`FEAT-NNN`** tag matching the `featureId`.

For example, tests targeting feature `FEAT-004` should be named following the pattern:
- `user-auth.FEAT-004.test.js`
- `describe("FEAT-004: User Authentication", ...)`

This structure is entirely language-agnostic and enables the framework to report test coverage vertically against features rather than arbitrarily across files.
