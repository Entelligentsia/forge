# Generation Integrity Audit

Observations, analysis, and remediation guidelines for the Forge generation pipeline
and the command-to-workflow delegation pattern.

---

### 1.3 Dogfooding Instance: Spec Violations

The dogfooding instance (this project) was last generated at v0.8.10 and has two
spec violations:

**Missing `effort:` frontmatter.** The generation spec (`generate-commands.md`)
prescribes `effort: max | high | medium | low` for every command. None of the 13
generated commands include it. Without it, the model allocates reasoning budget
blindly — `/approve` (which should be `max`) gets the same treatment as `/collate`
(which should be `low`).

**MASTER_INDEX included despite explicit prohibition.** The spec says:

> **Do NOT include a `Read engineering/MASTER_INDEX.md` line in the command.**
> The workflow itself loads MASTER_INDEX as its first step — loading it in the
> command gives the model enough context to rationalise skipping the workflow
> entirely and doing the work inline. Workflow first, always.

Yet 11 of 13 generated commands include `Also read engineering/MASTER_INDEX.md for
current project state.` Only `approve.md` and `collate.md` omit it.

### 1.4 Non-Deterministic Generation

The same generation spec produces different output depending on which model ran
`/forge:init` or `/forge:regenerate`. Four distinct formats have been observed:

| Generated by | Frontmatter | Body Style | MASTER_INDEX | AUTO-GENERATED tag |
|---|---|---|---|---|
| Early (cartographer) | `model: sonnet/opus/haiku` | Thin: "Read X and follow it." | Included | No |
| Mid (spectral) | `effort: high` | Verbose: numbered step list | Included | No |
| Recent (emberglow) | `effort: high` | Thin: "Read X and follow it." | Included | Yes |
| Dogfooding | `description:` only | Thin: "Read X. Also read Y." | Included | No |

The generation spec (`generate-commands.md`) is prose instructions to the model.
The model interprets them loosely — sometimes adding `effort:`, sometimes not;
sometimes writing verbose step lists, sometimes thin wrappers; sometimes including
MASTER_INDEX despite being told not to.

### 1.5 Config Format Drift

Spectral's config is structurally different from the current schema:

```jsonc
// Spectral (old format — missing fields)
{ "installedSkills": [...], "paths": {...}, "stack": {...}, "processes": {...}, ... }

// Current format (dogfooding, emberglow)
{ "version": "1.0", "project": {...}, "stack": {...}, "commands": {...}, "paths": {...}, "pipeline": {...}, "installedSkills": [...] }
```

Spectral is missing `version`, `project`, `commands`, and `pipeline`. It was
initialized with an older config schema and never migrated to the current format.

## 2. Analysis

### 2.1 The Delegation Pattern: Sound in Principle

The thin-wrapper pattern — a command that delegates to a workflow file — is
architecturally sound:

- **Single source of truth**: The workflow file is the authority; the command is
  just a pointer.
- **Independent regeneration**: Commands and workflows can be regenerated
  independently, with idempotency checks.
- **Ergonomic namespacing**: Project commands are bare (`/plan`, `/implement`),
  infrastructure commands are namespaced (`/forge:init`, `/forge:update`).

The semantic split is correct: `forge:*` means "manage the Forge system", bare
means "execute project work". Users run `/plan` 20 times a day and `/forge:update`
once a month.

### 2.2 The Delegation Pattern: Broken in Practice

The pattern depends on the model faithfully following the instruction "Read
`.forge/workflows/plan_task.md` and follow it." This is a request, not an
enforcement mechanism. The model is a reasoning agent that optimises for
efficiency. Given the full project state (from MASTER_INDEX), the task arguments,
and prior conversation context, the model can shortcut the workflow when it
believes it already has enough information.

The generation spec identified this risk explicitly:

> Loading [MASTER_INDEX] in the command gives the model enough context to
> rationalise skipping the workflow entirely and doing the work inline.

But every instance across the fleet violates this prohibition. The risk is not
theoretical — it's the most likely explanation for why agents sometimes bypass
workflow steps.

### 2.3 Why Generation Is Non-Deterministic

The generation prompts (`forge/init/generation/generate-commands.md` and
similar) are prose instructions. The model fills in content based on its
interpretation, which varies with model version, context window, temperature,
and surrounding prompt structure. There is no template, no schema, and no
post-generation validation.

Each `/forge:init` or `/forge:regenerate` run produces slightly different output
because:

1. The model decides how much to include in the command body (thin wrapper vs.
   verbose step list).
2. The model decides whether to include `effort:` frontmatter even when the spec
   prescribes it.
3. The model decides whether to include MASTER_INDEX even when the spec prohibits it.
4. The model decides the naming convention for workflow files (old `engineer_*`
   vs. new `plan_task.md`).
5. Different model versions produce different `model:` vs `effort:` frontmatter.

### 2.4 The Idempotency Check Only Prevents Overwrites

The generation spec's idempotency check is well-designed for its stated purpose:
don't clobber user-customized commands. But it cannot detect:

- **Spec violations**: A command that includes MASTER_INDEX or lacks `effort:`
  passes the idempotency check because it "references `.forge/workflows/` and
  the workflow file exists."
- **Stale pointers**: Cartographer's broken commands reference `.forge/workflows/`
  paths that don't exist — but the check only verifies the pointer format, not
  whether the target file is current.
- **Format drift**: A command with `model: sonnet` frontmatter passes the check
  because it references a workflow path, even though the spec now says to use
  `effort:` instead.

### 2.5 Migration Gaps

The `migrations.json` chain specifies `regenerate` targets per version. The
Store Custodian migration (0.8.10 → 0.9.0) lists `workflows` and `skills` but
not `commands`. This means `/forge:update` with this migration would regenerate
workflows (renaming `engineer_plan_task.md` to `plan_task.md`) but leave the
command wrappers pointing at the old names — exactly the Cartographer breakage.

## 3. Guidelines for Remediation

### 3.1 Immediate: Fix the Dogfooding Instance

The dogfooding instance (this project) has two correctable spec violations:

1. **Add `effort:` frontmatter** to all 13 commands per the spec table:

   | Command | effort |
   |---|---|
   | `review-plan.md` | `max` |
   | `review-code.md` | `max` |
   | `approve.md` | `max` |
   | `plan.md` | `high` |
   | `implement.md` | `high` |
   | `fix-bug.md` | `high` |
   | `sprint-plan.md` | `high` |
   | `run-task.md` | `high` |
   | `run-sprint.md` | `high` |
   | `sprint-intake.md` | `high` |
   | `retrospective.md` | `medium` |
   | `collate.md` | `low` |
   | `commit.md` | `low` |

2. **Remove MASTER_INDEX from command bodies.** The workflow loads it as its
   first step. Including it in the command pre-loads context that the model uses
   to rationalise skipping the workflow.

The canonical command format after remediation:

```markdown
---
effort: high
---
<!-- AUTO-GENERATED by /forge:init -->
Read `.forge/workflows/plan_task.md` and follow it exactly.
Task ID: $ARGUMENTS
```

No MASTER_INDEX. No step lists. No description line. The `effort:` frontmatter
and `<!-- AUTO-GENERATED -->` tag are the only additions beyond the delegation
line and arguments.

### 3.2 Immediate: Add Commands to Migration Regenerate Targets

Every migration that renames or restructures workflows must also list `commands`
in its `regenerate` array. The v0.8.10 → v0.9.0 migration should have included
`commands` alongside `workflows` and `skills`.

**Rule**: If a migration changes workflow filenames, it must regenerate commands.
If a migration changes command frontmatter conventions (e.g., `model:` → `effort:`),
it must regenerate commands.

### 3.3 Short-Term: Post-Generation Validation in `/forge:health`

`/forge:health` should check for and report:

1. **Broken workflow pointers**: Every command that references a `.forge/workflows/`
   path must resolve to an existing file.
2. **Missing `effort:` frontmatter**: Every generated command should have `effort:`
   with a valid value (`max`, `high`, `medium`, `low`).
3. **Prohibited content**: Commands should not contain a MASTER_INDEX read line.
4. **Stale `forgeRoot`**: The `forgeRoot` in `config.json` should match the
   currently installed plugin version.
5. **Config schema drift**: `config.json` should contain all required fields
   (`version`, `project`, `commands`, `paths`, `pipeline`, `installedSkills`).
6. **Deprecated frontmatter**: Commands should not use `model:` frontmatter.

The health check should produce a clear, actionable report:

```
Forge Health: Generation Integrity
════════════════════════════════════
✗ plan.md: missing effort: frontmatter
✗ plan.md: contains prohibited MASTER_INDEX read
✗ 6 commands point to missing workflow files
✓ forgeRoot points to current version (0.9.2)
✓ config.json has all required fields
```

### 3.4 Medium-Term: Deterministic Command Generation

Replace the prose-based command generation with a **template** that the model
fills in, not writes from scratch.

The canonical command template:

```markdown
---
effort: {{effort}}
---
<!-- AUTO-GENERATED by /forge:init -->
Read `.forge/workflows/{{workflow_filename}}` and follow it exactly.
{{argument_line}}
```

Where:
- `{{effort}}` comes from a fixed mapping (not model interpretation)
- `{{workflow_filename}}` comes from the list of generated workflow filenames
- `{{argument_line}}` is `Task ID: $ARGUMENTS`, `Sprint ID: $ARGUMENTS`, etc.

The model's job becomes filling variables, not writing the file format. This
eliminates the non-determinism that produces four different command formats
across four projects.

The generation spec should be updated to prescribe the template explicitly,
with an example that shows the exact output format. The idempotency check
should verify against the template structure, not just the workflow pointer.

### 3.5 Medium-Term: Post-Generation Validation Phase

Add a validation step to `/forge:init` and `/forge:regenerate` after command
generation (Phase 7). This step should:

1. **Schema-validate each command** against the canonical template.
2. **Resolve every workflow pointer** to confirm the target file exists.
3. **Check for prohibited content** (MASTER_INDEX reads, `model:` frontmatter).
4. **Verify config.json** matches the current schema.

If validation fails, regenerate the offending files rather than leaving them
in a broken state. If regeneration still fails, flag the issue and continue
rather than blocking the entire init.

### 3.6 Long-Term: Config Migration in `/forge:update`

When `/forge:update` detects that `config.json` uses an older schema (missing
`version`, `project`, `commands`, or `pipeline` fields), it should offer to
migrate the config to the current format rather than leaving it in a stale
state. This is analogous to how migrations handle workflow renames — the update
should bring all generated artifacts forward to the current schema.

The config migration should:
1. Preserve all existing fields.
2. Add missing required fields with sensible defaults.
3. Restructure flat fields into nested objects where the schema has changed
   (e.g., `stack.primaryLanguage` → `stack.languages`).
4. Update `forgeRoot` to point to the currently installed plugin version.
5. Record the migration in the config for traceability.

## 4. Command Delegation: Design Rationale

### Why Commands Delegate to Workflows

The thin-wrapper pattern exists for three reasons:

1. **Single source of truth.** The workflow file contains the complete algorithm —
   persona identity, context loading sequence, step-by-step instructions,
   knowledge writeback, event emission. The command is just a pointer. If the
   workflow changes, the command doesn't need updating.

2. **Ergonomic namespacing.** Project commands are bare (`/plan`) because they're
   used dozens of times per day. Infrastructure commands are namespaced
   (`/forge:init`) because they're used infrequently. This split is reflected in
   their locations: project commands in `.claude/commands/`, plugin commands in
   `forge/commands/`.

3. **Regeneration independence.** Commands and workflows can be regenerated
   independently. The idempotency check in `generate-commands.md` ensures that
   a command wrapper is only overwritten if its workflow pointer is stale or
   missing, not if it has been customised by the user.

### Why MASTER_INDEX Must Not Be in Commands

The workflow's Step 1 is always "Load Context" — it reads MASTER_INDEX, the
architecture index, and the relevant task prompt. This controlled context
loading is deliberate:

- The model starts with only the workflow instructions and task ID.
- It reads MASTER_INDEX only when the workflow tells it to.
- This ensures the model processes the workflow steps in order.

If MASTER_INDEX is pre-loaded in the command, the model has the full project
state map before it reads the workflow. This gives it enough context to
rationalise skipping the workflow entirely and doing the work inline — defeating
the entire delegation pattern.

### Why `effort:` Matters

The `effort:` frontmatter is a model-agnostic signal for reasoning budget
allocation. Without it:

- `/approve` (architectural gate, needs deepest reasoning) gets the same
  treatment as `/collate` (mechanical markdown regeneration).
- The model has no hint about how much context window to reserve for reasoning
  vs. reading.
- Token-heavy workflows (plan, review) may truncate their reasoning to fit
  within default allocation.

The mapping is fixed and should not be left to model interpretation:

| Level | Use For | Reasoning Budget |
|-------|---------|-----------------|
| `max` | Architectural gates (review, approve) | Maximum available |
| `high` | Design and implementation work | Substantial |
| `medium` | Reflection and synthesis | Moderate |
| `low` | Mechanical operations (commit, collate) | Minimal |

### Why Collision Risk Is Acceptable (For Now)

The bare command names (`/plan`, `/commit`, `/review-code`) are generic enough
that other plugins could claim them. However:

- The collision requires another plugin that defines the exact same filenames.
- Forge's idempotency check won't overwrite existing non-Forge files.
- The user sees a clear error when two plugins define the same command.

This risk should be documented but not over-engineered. If collisions become
common, the mitigation is to generate namespaced aliases (`/forge-plan`) as
additional entry points alongside the bare names.

## 5. `meta/` Classification Review

The `meta/` directory contains Forge's core intellectual property — templates
that are read during `/forge:init` and `/forge:regenerate` to produce
project-specific artifacts. They are not runtime components discoverable by
Claude Code's plugin system.

### Current Classification

```
meta/
  personas/          Agent role definitions (8 files)
  skills/            Skill generation templates (7 files)
  workflows/         Workflow generation templates (16 files)
  templates/         Document structure definitions (8 files)
  tool-specs/        Tool generation specifications (6 files)
  store-schema/      JSON data model (5 files)
  skill-recommendations.md  Marketplace skill mapping
```

### Naming Collision Risks

| Directory | Collision Risk | Assessment |
|-----------|---------------|------------|
| `meta/skills/` | **High** — Claude Code expects `skills/` to contain runtime `SKILL.md` files in subdirectories | Confusingly named; these are skill *templates*, not auto-activated skills |
| `meta/personas/` | **Medium** — Claude Code expects `agents/` for agent definitions | Could be confused with agent auto-selection |
| `meta/workflows/` | **Low** — No standard Claude Code `workflows/` convention yet | Future-proofing concern |
| `meta/templates/` | **Medium** — Everything in `meta/` is a template | Semantic overlap with other template types |
| `meta/tool-specs/` | **Low** — No standard convention | Clear enough in context |
| `meta/store-schema/` | **None** — Clear and specific | Good as-is |

### Recommended Renames

| Current | Proposed | Why |
|---------|----------|-----|
| `meta/skills/` | `meta/skill-templates/` or `meta/capabilities/` | Eliminates naming collision with Claude Code's `skills/` auto-discovery; clarifies these are generation inputs, not runtime components |
| `meta/templates/` | `meta/document-formats/` or `meta/output-schemas/` | Disambiguates from the fact that everything in `meta/` is a template of some kind |
| `meta/personas/` | `meta/role-definitions/` | More precise; eliminates implication that these are Claude Code agents |
| `meta/workflows/` | `meta/workflow-templates/` | Future-proofing against a potential Claude Code `workflows/` convention |

These renames affect the generation prompts in `init/generation/` and the
workflow files themselves (which reference `meta/personas/` paths). A rename
must be accompanied by updates to all references.

## 6. Plugin Settings Accuracy

### Current `plugin.json`

| Field | Value | Assessment |
|-------|-------|-----------|
| `name` | `forge` | Correct, kebab-case |
| `version` | `0.9.2` | Matches latest migration chain |
| `description` | Present | Accurate |
| `license` | `MIT` | Present |
| `author` | Complete (name, email, url) | Good |
| `repository` | Correct GitHub URL | Good |
| `updateUrl` | Points to main branch | Correct for direct installs |
| `migrationsUrl` | Points to main branch | Correct |
| `keywords` | 5 relevant terms | Good |

### Missing but Recommended

- **`homepage`**: No documentation site URL. The repository URL serves double duty.
  Consider adding a distinct homepage URL (e.g., to the Forge documentation site
  or README).
- **No custom path declarations**: The plugin relies on auto-discovery for
  `commands/` and `hooks/`. This works for standard directories. The
  non-standard directories (`meta/`, `init/`, `tools/`, `schemas/`, `vision/`)
  are not auto-discovered, which is correct — they are read by command
  invocations, not by Claude Code's component scanner.

### Not Missing (Correctly Absent)

- **No `agents`**: Forge doesn't register Claude Code agents. Personas are
  templates for generation, not runtime agents.
- **No `skills`**: Forge doesn't ship runtime skills. Meta-skills are templates.
- **No `mcpServers`**: Forge doesn't ship MCP servers.

### `hooks.json`

- SessionStart and PostToolUse hooks are properly configured.
- `${CLAUDE_PLUGIN_ROOT}` is used for portable paths.
- Timeouts are appropriate (10s for update check, 5s for error triage).

### Opportunity: Runtime Skills

Forge could register runtime skills in a standard `skills/` directory that
activate based on context. For example:

- A skill that activates when the model detects `.forge/store/` activity, providing
  store custodian knowledge.
- A skill that activates during plan or review phases, providing SDLC guidance.

This would give Forge visibility in the coding agent session beyond the 10
explicit `forge:*` commands, without requiring the user to invoke a command.
Currently, the only runtime visibility is through the generated project commands
in `.claude/commands/`.

## 7. Checklist: Version Bump Requirements

When bumping the Forge version, the following must be verified:

- [ ] `plugin.json` version updated
- [ ] `migrations.json` entry added with correct `from`, `version`, `notes`
- [ ] `migrations.json` `regenerate` list includes `commands` if workflows were
      renamed or command format conventions changed
- [ ] Security scan run on `forge/` source directory
- [ ] Scan report saved to `docs/security/scan-v{VERSION}.md`
- [ ] Security table in README.md updated
- [ ] Dogfooding instance updated via `/forge:update` and `/forge:regenerate`
- [ ] Testbench projects updated and verified
- [ ] `forgeRoot` in testbench configs matches installed plugin version
- [ ] All generated command wrappers have valid workflow pointers
- [ ] All generated commands include `effort:` frontmatter
- [ ] No generated commands include MASTER_INDEX reads
- [ ] `config.json` in all instances matches current schema
