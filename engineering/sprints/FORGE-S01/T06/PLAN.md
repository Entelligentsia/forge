# PLAN — FORGE-S01-T06: Retrospective meta-workflow — cost analysis and baselines

**Task:** FORGE-S01-T06
**Sprint:** FORGE-S01
**Estimate:** M

---

## Objective

Extend `forge/meta/workflows/meta-retrospective.md` (and by extension the generated
`.forge/workflows/sprint_retrospective.md`) to incorporate token cost analysis into
the sprint retrospective. The extension adds: (1) cost-data loading and aggregation
in Step 2, (2) a "Cost Analysis" section in the retrospective output (Step 5), and
(3) a new step that writes/updates `.forge/store/COST_BASELINES.json` with rolling
medians for future sprint comparisons.

## Approach

### What is being changed

The only `forge/` artifact touched is the meta-workflow definition:
`forge/meta/workflows/meta-retrospective.md`. The generated workflow
`.forge/workflows/sprint_retrospective.md` is regenerated from this meta-workflow
by the `/forge:regenerate` command (users run this after installing the new plugin
version via `/forge:update`).

No CJS tools are added or modified. No schemas change. Token data is read from the
existing `.forge/store/events/{SPRINT_ID}/` directory (already loaded in Step 1)
using the fields added in T01 (`inputTokens`, `outputTokens`, `cacheReadTokens`,
`cacheWriteTokens`, `estimatedCostUSD`, `tokenSource`). The `COST_REPORT.md`
generated by collate (T05) is also available as a pre-computed view — Step 2 can
read it to avoid re-computing the per-task breakdown.

### Changes to `forge/meta/workflows/meta-retrospective.md`

**Step 1 — Load Sprint Data (extend):**

Add an instruction to also read `COST_REPORT.md` from the sprint directory (if it
exists). If it does not exist (T05 not yet run, or sprint has no token data), note
that cost analysis will be skipped.

**Step 2 — Analyse Patterns (extend):**

Add a "Cost Analysis" subsection with the following instructions:

- From the loaded events (skip events with no `inputTokens` field), compute:
  - Total tokens (input + output) and total `estimatedCostUSD` for the sprint
  - Per-task token totals, sorted descending by total tokens (most expensive first)
  - Review overhead ratio = sum of tokens for review-role events / total sprint tokens
    (review roles: `supervisor`, `architect` in review/approve phases)
  - If `.forge/store/COST_BASELINES.json` exists, compare the current sprint median
    tokens per estimate size (S/M/L/XL) to the stored baselines and flag any task
    that consumed >2x the baseline for its estimate tier.
- Gracefully skip all cost aggregation if no events have token fields.

**Step 5 — Write Retrospective (extend):**

Add an instruction to include a "Cost Analysis" section in `RETROSPECTIVE.md` with:

```markdown
## Cost Analysis

| Metric | Value |
|---|---|
| Total input tokens | {N} |
| Total output tokens | {N} |
| Total estimated cost | ${X.XX} |
| Review overhead ratio | {X%} |

### Most Expensive Tasks

| Task | Total Tokens | Est. Cost USD | Source |
|---|---|---|---|
| {TASK_ID} | {N} | ${X.XX} | reported / estimated / mixed |

### Baseline Comparison

| Estimate | Baseline Median Tokens | This Sprint Median | Delta |
|---|---|---|---|
| S | {N} | {N} | {±%} |

_Baseline comparison omitted — no prior baseline data._
```

If no token data is available, emit a single line:
`_No token data available for this sprint._`

**New Step 5.5 — Update Cost Baselines (new step, between Step 5 and Step 6):**

Compute rolling baselines across **all** sprints (not just the current one) by:

1. Reading all event files from `.forge/store/events/` (all sprint subdirectories),
   filtering to events with `inputTokens` defined.
2. Joining each event with its task record (from `.forge/store/tasks/{taskId}.json`)
   to obtain the task's `estimate` field (S/M/L/XL).
3. For each estimate tier, compute the median of `(inputTokens + outputTokens)`
   across all completed tasks in that tier that have token data.
4. Compute the median review overhead ratio across all sprints.
5. Write (or overwrite) `.forge/store/COST_BASELINES.json`:

```json
{
  "medianTokensPerEstimate": {
    "S": <number>,
    "M": <number>,
    "L": <number>,
    "XL": <number>
  },
  "medianReviewOverhead": <number>,
  "sampleSize": <integer>,
  "lastUpdated": "<ISO timestamp>"
}
```

Only include tiers for which at least one data point exists. If no token data
exists across any sprint, skip writing the file entirely (do not write an empty
or zero-filled baseline).

**Step 6 — Collate (unchanged):** remains as-is.

### RETROSPECTIVE_TEMPLATE.md (extend)

Add the "Cost Analysis" section stub to `.forge/templates/RETROSPECTIVE_TEMPLATE.md`
after "Bug Pattern Analysis", so future retrospectives include the section by default.

### Generated workflow

After modifying the meta-workflow and template, regenerate
`.forge/workflows/sprint_retrospective.md` to reflect the changes (this is what
users see in their installed project). The regeneration uses the standard workflow
generation pattern — apply the same changes that would be produced by
`/forge:regenerate workflows`.

## Plugin Impact Assessment

- **Version bump required?** Yes — bundled with T08. Changes to
  `forge/meta/workflows/meta-retrospective.md` are material (users who regenerate
  workflows will receive a different retrospective procedure).
- **Migration entry required?** Yes (in T08) — `regenerate: ["workflows"]` so users
  run `/forge:update` → `/forge:regenerate workflows` to receive the updated retrospective
  workflow. New version is `0.4.0` (as planned in T01).
- **Security scan required?** Yes — any change to `forge/` requires a scan. Bundled
  with T08.
- **Schema change?** No — `COST_BASELINES.json` is a project-internal file, not a
  validated schema record. No `forge/schemas/*.schema.json` files are touched.

- **Store validation:** `node forge/tools/validate-store.cjs --dry-run` — run to confirm
  no regressions (no schema files changed, but guards against accidental drift).
- **Manual review:** Read the updated `meta-retrospective.md` and verify:
  - Step 1 instructs loading `COST_REPORT.md`
  - Step 2 has a "Cost Analysis" subsection with the three computed metrics

- **Generated workflow review:** Read the regenerated `.forge/workflows/sprint_retrospective.md`
  and confirm all new instructions from the meta-workflow are present.

- **Distribution:** Users must run `/forge:update` after T08 ships (version `0.4.0`),
  then `/forge:regenerate workflows` to receive the updated retrospective workflow in their
