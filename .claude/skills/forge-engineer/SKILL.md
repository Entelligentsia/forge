# Forge Engineer

Implement approved changes to the Forge plugin source (`forge/`).

## Iron Laws

1. ONLY edit files in `forge/`. NEVER edit `.forge/` or `engineering/` to fix Forge itself.
2. Every change to a `.cjs` tool must be preceded by a failing test.
3. Schema changes that affect entity lifecycle require concepts diagram updates.
4. Version bump required for material changes. Migration entry required. Tests must pass.
5. Silent continuation past failures is never acceptable.

## Two-Layer Boundary

```
forge/          ← PLUGIN SOURCE. You work here.
.forge/         ← GENERATED OUTPUT. Never edit directly.
engineering/    ← DOGFOODING KB. Never edit as part of plugin work.
```

If you catch yourself editing `.forge/workflows/`, `.forge/personas/`, or
`.forge/skills/` — stop. The fix goes in `forge/meta/`. The generated output
is regenerated by users via `/forge:update`.

## Implementation Rules

### For `.cjs` tool modifications — test-first, always

1. **Write a failing test** in `forge/tools/__tests__/<tool>.test.cjs` that exposes the bug or defines the new behaviour
2. **Run the test suite** — confirm the new test fails: `node --test forge/tools/__tests__/<tool>.test.cjs`
3. **Implement the fix/feature** in `forge/tools/<tool>.cjs`
4. **Run the test suite again** — confirm the new test passes
5. **Run the full suite** — confirm no regressions: `node --test forge/tools/__tests__/*.test.cjs`

No exceptions. This is the CLAUDE.md rule. Every change to a `.cjs` script
must be preceded by a failing test. New exports added to `module.exports` in
any tool require at least one test in the matching test file.

For **refactors**: run existing tests before touching code → refactor → run tests
again — they must still pass. If behaviour changes intentionally, write the test
for the new behaviour first.

### For `meta/` workflow modifications — edit the meta source

- Edit `forge/meta/workflows/meta-*.md`, NOT `.forge/workflows/*.md`
- Edit `forge/meta/personas/meta-*.md`, NOT `.forge/personas/*.md`
- Edit `forge/meta/skills/meta-*.md`, NOT `.forge/skills/*.md`

If adding a new meta file, update the mapping table in
`forge/tools/build-manifest.cjs` and run:
```sh
node forge/tools/build-manifest.cjs --forge-root forge/
```

### For `hooks/` modifications

- Edit `forge/hooks/*.js` directly
- If adding a new hook, update `forge/hooks/hooks.json`
- Test the hook logic by reading the test file in `forge/hooks/__tests__/`

### For schema modifications

- Edit `forge/schemas/*.schema.json` directly
- If the schema change affects entity lifecycle (status enums, state machinery
  of Sprint, Task, Bug, or Feature entities), update the corresponding
  handwritten state diagram in `docs/concepts/*.md`
- Also update `forge/meta/store-schema/*.schema.md` documentation

### For command modifications

- Edit `forge/commands/*.md` directly
- Commands use `!` backtick-exec for `$FORGE_ROOT` resolution:
  ```
  FORGE_ROOT: !`echo "${CLAUDE_PLUGIN_ROOT}"`
  ```

## Stack Constraints

- Node.js 18+ CJS scripts only. No npm dependencies. Built-ins: `fs`, `path`, `os`,
  `https`, `readline`. Every hook and tool must be self-contained.
- Introducing `require()` of any non-built-in module is absolutely blocked.
- Paths must come from `.forge/config.json`, never hardcoded as `'engineering/'` or
  `'.forge/store/'` string literals in tool code.

## Hook Discipline

Every hook (`forge/hooks/*.js`) MUST have:
- `'use strict';` as the first non-comment line
- `process.on('uncaughtException', () => process.exit(0))` — hooks MUST NEVER exit non-zero

## Tool Discipline

Every tool (`forge/tools/*.cjs`) MUST have:
- `'use strict';` as the first line
- A top-level try/catch that calls `process.exit(1)` on error
- `--dry-run` flag honoured before any writes

## Git Protocol

- Stage specific files by path — never `git add -A` or `git add .`
- Commit message prefixes: `feat:`, `fix:`, `docs:`, `chore:`, `security:`, `release:`
- Always include `Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>`
- Never `--no-verify`. If a hook fails, diagnose and fix, then create a NEW commit

## Common Pitfalls

- Introducing `require()` of any non-built-in module (absolutely blocked)
- Hardcoding `'engineering/'` or `'.forge/store/'` as string literals in tools
- Editing `.forge/workflows/`, `.forge/personas/`, or `.forge/skills/` directly
  (regenerated output — fix goes in `forge/meta/`)
- Forgetting to update `README.md` security table when adding `docs/security/scan-v*.md`

## Syntax Verification

After every JS/CJS edit, run:

```sh
node --check <file>
```

This catches syntax errors before test execution. Do this before running tests.

For schema changes, also run:

```sh
node forge/tools/validate-store.cjs --dry-run
```

This validates the schema without writing.

## Progressive Verification

After each logical group of changes:

```sh
node --test forge/tools/__tests__/*.test.cjs
```

Do NOT batch all changes and test only at the end. Test incrementally.

If tests fail, fix before continuing. Never push forward with a failing test.

## What Constitutes a Material Change

A change is material (requires version bump) if it can affect the engineering
workflow of a user who has already installed Forge:

- Bug fixes to any command, hook, tool spec, or workflow
- Changes to tool specs that alter generated tool behaviour
- Changes to command files that alter how commands behave
- Changes to hooks
- New commands or tools
- Schema changes to `forge/schemas/`

Not material: documentation-only changes, typo fixes in `docs/`, README updates.
