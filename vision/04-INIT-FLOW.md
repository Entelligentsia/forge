# 04 — The Init Flow

`/forge:init` transforms an existing codebase into a fully operational SDLC instance in 9 phases.

---

## The Promise

A developer with an existing codebase runs one command and gets a complete engineering practice — knowledge base, agent personas, workflows, templates, tools, and slash commands — tailored to their project. Under an hour to first sprint, including human review.

---

## Phase 1 — Discover

Five parallel discovery prompts scan the project using Claude Code's built-in tools:

| Prompt | Scans | Tools Used | Produces |
|--------|-------|-----------|----------|
| `discover-stack.md` | package.json, requirements.txt, go.mod, Cargo.toml, Dockerfile, Makefile | Glob, Read | Language, frameworks, versions, runtime |
| `discover-processes.md` | docker-compose.yml, Procfile, systemd units, ecosystem.config.js, CI config | Glob, Read | Service topology, build/deploy commands, process boundaries |
| `discover-database.md` | ORM models, schema files, migration dirs, seed scripts | Grep, Read | Entity inventory, relationships, field types |
| `discover-routing.md` | Route definitions, middleware, auth decorators, API docs | Grep, Read | API surface map, auth strategy, public vs protected routes |
| `discover-testing.md` | Test directories, CI config, test scripts, coverage config | Glob, Read, Bash | Test framework, test command, build command, lint command |

**Output**: `.forge/config.json` + an internal discovery context document.

### Example: .forge/config.json (generated)

```json
{
  "version": "1.0",
  "project": {
    "prefix": "ACME",
    "name": "Acme Analytics",
```

## Phase 2 — Generate Knowledge Base

Reads the discovery context and generates architecture + business domain documentation.

### What Gets Generated

```
.forge/
  store/                  JSON store directories (empty, with .gitkeep)
    sprints/
    tasks/
    bugs/
    events/

engineering/
  architecture/
    INDEX.md              Links to sub-docs, describes what each covers
    stack.md              Django 4.2, DRF 3.15, React 18, PostgreSQL 15, Celery 5
    processes.md          Gunicorn workers, Celery worker, Celery beat, React dev server
    database.md           47 Django models, key relationships, migration patterns
    routing.md            89 DRF views, URL namespaces, auth decorator usage
    deployment.md         Docker Compose services, GitHub Actions CI/CD pipeline
  business-domain/
    INDEX.md              Entity list with one-line descriptions
    entity-model.md       Workspace, Team, User, Dashboard, Subscription, Invoice...
  stack-checklist.md      Initial review checklist (5-10 items from auth/framework detection)
  MASTER_INDEX.md         Empty scaffold
  sprints/                Task work artifact directories (empty scaffold)
  bugs/                   Bug work artifact directories (empty scaffold)
```

### Confidence Ratings

Every generated doc includes a header:

```markdown
<!-- AUTO-GENERATED by /forge:init — confidence: 75%
     Review and correct before first sprint.
     Lines marked [?] need human verification. -->
```

The `[?]` marker flags specific lines the LLM was uncertain about:

```markdown
## Workspace Entity
- `id` (UUID, primary key)
- `name` (CharField, max_length=255)
- `owner` (ForeignKey → User)
- `plan` (CharField) [?] — choices unclear, possibly ['free', 'pro', 'enterprise']
- `created_at` (DateTimeField, auto_now_add)
```

## Phase 3 — Generate Personas

Reads: meta-personas + discovery context + generated knowledge base.
Generates: persona context that gets embedded into each workflow.

The persona is not a separate file — it's the opening section of each workflow that establishes the agent's identity, knowledge, and constraints.

### What Changes Between Projects

| Aspect | WalkInto Engineer | Maya's Engineer |
|--------|-------------------|-----------------|
| Stack context | "Node.js/Express, MongoDB, Jade, PM2" | "Django 4.2/DRF, PostgreSQL, React 18, Celery" |
| Data access | "MongoDB through routes/db/ wrapper" | "Django ORM querysets, never raw SQL" |
| Test command | "node routes/Tests/<file>.js" | "pytest && npm test" |
| Build command | "grunt build" | "npm run build" |
| Auth check | "user.isLoggedIn middleware" | "@login_required on views, IsAuthenticated on API" |
| Entities | "tours, users, userpanos, products" | "Workspace, Team, Dashboard, Subscription" |

## Phase 4 — Generate Templates

Reads: meta-templates + discovery context + knowledge base.
Generates: `.forge/templates/` with project-specific document templates.

### How a Meta-Template Becomes Project-Specific

**Meta-Plan** defines sections: Objective, Approach, Files, Data Model, Testing, Acceptance Criteria, Operational Impact.

**Maya's PLAN_TEMPLATE.md** adds:
- "Django Models" subsection under Data Model (because Django was detected)
- "DRF Serializers/Views" subsection (because DRF was detected)
- "React Components" section (because React was detected)
- "Celery Tasks" section (because Celery was detected)
- "Database Migrations" section with `makemigrations --check` verification
- Test evidence section expects `pytest` + `jest` output format

A Go project's PLAN_TEMPLATE.md would have none of these — it would have "Go Packages", "Interface Changes", "go vet / go test Output" instead.

## Phase 5 — Generate Atomic Workflows

Reads: meta-workflows + discovery context + knowledge base + generated personas + generated templates.
Generates: `.forge/workflows/` — complete, project-specific workflow files.

### The 14 Core Workflows

| Generated Workflow | From Meta-Workflow | Project-Specific Elements |
|-------------------|-------------------|--------------------------|
| `engineer_plan_task.md` | `meta-plan-task.md` | Architecture sub-docs by name, entity names, template path |
| `engineer_implement_plan.md` | `meta-implement.md` | Syntax check, test command, build command, stack conventions |
| `engineer_update_plan.md` | `meta-update-plan.md` | Same as plan_task |
| `engineer_update_implementation.md` | `meta-update-implementation.md` | Same as implement |
| `engineer_fix_bug.md` | `meta-fix-bug.md` | Bug store path, domain docs, test commands |
| `engineer_commit_task.md` | `meta-commit.md` | Store paths, ID format, co-author line |
| `supervisor_review_plan.md` | `meta-review-plan.md` | Architecture sub-docs, stack conventions, checklist loading |
| `supervisor_review_implementation.md` | `meta-review-implementation.md` | Stack checklist, code convention checks, framework-specific items |
| `architect_sprint_plan.md` | `meta-sprint-plan.md` | ID format, entity names, operational impact categories |
| `architect_approve.md` | `meta-approve.md` | Store paths, status values |
| `architect_review_sprint_completion.md` | (part of meta-sprint-plan) | Sprint artifact paths |
| `collator_agent.md` | `meta-collate.md` | Store paths, project description, index format |
| `sprint_retrospective.md` | `meta-retrospective.md` | Domain docs, stack checklist, workflow paths |
| `quiz_agent.md` | (lightweight, mostly generic) | Knowledge base paths |

## Phase 6 — Generate Orchestration

Reads: `meta-orchestrate.md` + generated atomic workflows.
Generates: `.forge/workflows/orchestrate_task.md`

The orchestrator wires the atomic workflows into a pipeline:

```
Plan → Review Plan → [loop max 3] → Implement → Review Code → [loop max 3] → Approve → Commit
```

Project-specific elements:
- Workflow paths (references the generated atomic workflows by name)
- Gate conditions (test commands, file existence checks)
- Model selection per role (Sonnet for Engineer, Opus for Supervisor/Architect, Haiku for Commit)
- Escalation format (uses project prefix in task IDs)

---

## Phase 7 — Generate Commands

Reads: generated workflows.
Generates: `.claude/commands/` — one slash command per workflow entry point.

Each command is simple — load the workflow, pass the argument. But project-specific:

```markdown
---
model: sonnet
---
Execute the Engineer Plan Task workflow for the specified task.

Usage: /engineer ACME-S01-T01
The argument is the Task ID: $ARGUMENTS

Steps:
1. Read @.forge/workflows/engineer_plan_task.md — this is your workflow.
2. Read @engineering/MASTER_INDEX.md for context.
3. Follow the workflow exactly.
```

---

## Phase 8 — Generate Tools

Reads: tool specs + `.forge/config.json` (language, paths, prefix).
Generates: `engineering/tools/` — deterministic tools in the project's language.

| Tool Spec | Python Project | Node.js Project | Go Project |
|-----------|---------------|-----------------|------------|
| `collate.spec.md` | `collate.py` | `collate.js` | `collate.go` |
| `seed-store.spec.md` | `seed_store.py` | `seed-store.js` | `seed-store.go` |
| `validate-store.spec.md` | `validate_store.py` | `validate-store.js` | `validate-store.go` |

The generated tool is idiomatic for the stack — uses the project's standard library, follows its naming conventions, integrates with its test runner if relevant.

## Phase 9 — Smoke Test

Validates the generated instance before handing off to the developer.

### Checks

1. **Structural**: all expected files exist
   - `.forge/config.json`, architecture docs, business domain docs, stack checklist
   - All 14 workflows, all commands, all templates, all tools

2. **Referential**: internal links resolve
   - Orchestrator's workflow paths point to existing files
   - Templates referenced in workflows exist
   - Architecture INDEX.md links resolve

3. **Tool execution**: `engineering/tools/validate-store --dry-run` runs without error

4. **Template coherence**: templates reference entities that exist in entity-model.md

5. **Self-correction**: if a check fails
   - Log the specific failure
   - Attempt to regenerate the failing component
   - Re-run the failed check
   - If still failing, report to user with diagnostic

### Report

```
/forge:init complete

Knowledge Base:
  architecture/ — 5 docs (stack, processes, database, routing, deployment)
  business-domain/ — 2 docs (INDEX, entity-model) — 47 entities
  stack-checklist.md — 8 initial items

Generated Artifacts:
  14 workflows in .forge/workflows/
  15 commands in .claude/commands/
  7 templates in .forge/templates/
  3 tools in engineering/tools/ (Python)

Smoke Test: all checks passed (1 self-corrected: fixed template reference)

Confidence: ~60% — review engineering/ docs before first sprint.
Lines marked [?] need verification.

Next: review the knowledge base, then run /sprint-plan
```
